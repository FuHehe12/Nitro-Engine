# 氮气引擎 / Nitro Engine

[English](./README_EN.md)

> Claude Code skill for natural Markdown translation. AI-powered, multi-language, fluent output.

**版本 / Version**: `v1.0.0`

> 翻译即写作——不是转换，而是重新表达。

一个 Claude Code 技能，专注于将 Markdown 文档翻译成**自然流畅的译文**。

**支持任意语言对**。目前仅在英文→中文上测试验证，其他语言对欢迎社区测试反馈。

---

## "准确"的翻译有什么问题？

机器翻译很快。AI 翻译更快。但出来的东西读起来就像……翻译。

对比一下：

| 原文 | 典型翻译 | 这个技能追求的效果 |
|------|----------|-------------------|
| The fact that the system is widely adopted by developers suggests that it addresses a genuine need. | 系统被开发者广泛采用的事实表明它解决了真正的需求。 | 既然这么多开发者都在用，说明它确实解决了实际问题。 |
| It is important to note that this approach may not be suitable for all use cases. | 重要的是要注意，这种方法可能不适用于所有用例。 | 值得一提的是，这套方法并非放之四海而皆准。 |
| The analysis of data was conducted using Python. | 数据的分析是使用 Python 进行的。 | 我们用 Python 分析了数据。 |

三段"典型翻译"语法上都对。但读起来就是翻译腔——被动、欧化、僵硬。

这个技能走的是另一条路：**翻译即写作**。目标不是转换文字，而是让同样的意思在目标语言中找到自然的表达方式。

---

## 核心理念

> **翻译即写作**——用另一种语言重新表达。

不是转换。是重新表达。

好的翻译应该让读者感觉原文就是用目标语言写的。他们不该察觉到这是"搬过来"的东西。

这很难做到。这个技能是尝试用 AI 能力去接近这个理想的一次实验。

---

## 工作原理

### 三步流程

大多数翻译工具直接进入"转换"阶段。这个技能走的是另一条路：

**1. 重构**——在脑海中构建原文的完整图景
- 谁在说话？对谁说？在什么情境下？
- 技术文档的感觉是"专家在给你讲解"
- 博客文章的感觉是"朋友在分享经验"
- 这个"想象空间"决定了翻译质量的上限

**2. 对话**——做出判断
- 这个词在当前语境下*到底*是什么意思？
- 母语者会怎么自然地表达这个概念？
- 这些判断会被记录在术语表和翻译简报中

**3. 写作**——用目标语言重新表达
- 不是逐词替换
- 而是站在想象空间里，用目标语言说话

### 大规模一致性

长文档会被拆分成片段并行翻译。但不同于简单的并行处理：

- **翻译简报**在任何翻译开始之前，就确定了文档的语气、风格和关键决策
- **基调样本**让所有代理锚定到同一个风格基准
- **共享术语表**确保所有片段的术语一致

结果：一万行的文档读起来像一个人写的，不是十个人。

### 可定制，适配你的领域

不同领域有不同的术语规范。这个技能支持：

- **用户术语表**（`user-glossary.md`）—— 定义特定术语的译法，技能会在所有文档中一致遵循你的选择
- **翻译样本**（`user-samples.md`）—— 提供你喜欢的翻译范例，技能会以此为风格参考

随着使用，这些文件会逐渐积累，技能会越来越符合你所在领域的表达习惯。

### 地域感知（中文）

中文不只是中文。这个技能区分不同地区：

| 英文 | 大陆（zh-CN） | 台湾（zh-TW） | 香港（zh-HK） |
|------|--------------|---------------|---------------|
| software | 软件 | 軟體 | 軟體 |
| program | 程序 | 程式 | 程式 |
| network | 网络 | 網路 | 網路 |
| database | 数据库 | 資料庫 | 資料庫 |

在翻译简报中指定目标地区，技能会自动调整。

## 工作流程

```
Phase 0: 输入分析
├── 检测语言、评估规模、选择策略
│
Phase 1: 通读与准备
├── 生成术语表 (_glossary.md)
├── 生成翻译简报 (_translation_brief.md)
├── 如需则拆分大文件（>1000 行）
│
Phase 1.5: 基调锚定（仅大文件）
├── 翻译一个代表性段落作为风格参考
│
Phase 2: 翻译执行
├── 小文件：主代理直接翻译
├── 大文件：子代理并行翻译
│
Phase 3: 校验与合并
├── 格式检查、术语一致性
├── 合并片段、平滑边界
│
Phase 4: 输出清理
├── 将新术语写回用户术语表
├── 清理临时文件
```

## 快速开始

**前置条件**：[Claude Code](https://github.com/anthropics/claude-code) CLI 和 Coding 订阅套餐。

1. 将技能复制到 Claude Code 技能目录：

```bash
cp -r translate ~/.claude/skills/
```

2. 用自然语言调用：

```
将 path/to/document.md 翻译为中文。
```

或者翻译整个文件夹：

```
将 path/to/folder/ 下所有 Markdown 文件翻译为中文。
```

3. 翻译后的文件会出现在同目录下，带语言后缀（如 `document_zh.md`）。

## 重要说明

### Token 消耗

这个技能使用主代理 + 子代理进行并行翻译。是 **Token 杀手**。

- 长文档（>1000 行）会同时派发多个子代理
- 需要使用 **Claude Code**（这是一个 Claude Code 技能）
- 强烈建议使用 **Coding 订阅套餐**（Claude Max、GLM Coding 或类似套餐）
- 按量付费的 API 翻译大文档会非常昂贵

### 并发限制（GLM Coding 套餐）

高峰期 GLM Coding 套餐可能资源不足。开始翻译前：

1. 根据你的套餐等级，要求限制并发子代理数量：
   ```
   限制并发子代理数量不超过 N（根据你的套餐等级调整 N）。
   ```

2. 高峰期翻译超大文档时，要求拆分成更小的片段：
   ```
   拆分成更小的片段（如每段 300 行），而不是默认的 600 行。
   ```

这样可以避免"资源耗尽"错误，提高成功率。

### 模型兼容性

在 **GLM5** 上测试过。其他模型未测试。

- 理论上 Opus 的写作质量会更好
- 如果你在其他模型上测试过，欢迎反馈

### 翻译 PDF

这个技能只处理 Markdown。PDF 翻译方案：

1. 使用 [MinerU](https://github.com/opendatalab/MinerU) 将 PDF 转为 Markdown
2. 执行 `/clear` 清除上下文
3. 翻译转换后的 Markdown
4. 再次执行 `/clear`
5. 让 Claude 修复格式问题：

```
检查并修复 XXX_zh.md 的标题层级问题，并修复 Markdown 格式。
```

**为什么要 `/clear`？** 每一步都会产生大量上下文，清除可以保证最佳效果，避免上下文污染影响翻译质量。

MinerU 输出的文件常有标题层级错误和 Markdown 语法小问题，需要额外清理。

## 局限性

- **输入格式**：仅支持 Markdown 和纯文本。PDF、Word、网页等需先转为 Markdown。
- **语言覆盖**：支持任意语言对。目前仅在英文→中文上测试，其他语言对待社区验证。
- **代码块**：代码块内容保持原样（行内代码注释可翻译）。
- **质量上限**：技能提供结构和流程，但最终质量取决于底层模型的写作能力。

## 背景

标准机器翻译的输出往往"准确但难读"——语法正确，却僵硬不自然。问题不在于准确性，而在于翻译被当作词汇转换，而非重新表达。

这个技能采用不同的理念：翻译即写作。目标是产出读起来像目标语言原生写作的文本。

基于 Claude Code 构建，利用代理能力在长文档中保持一致性，同时尊重领域特定的术语和风格偏好。
